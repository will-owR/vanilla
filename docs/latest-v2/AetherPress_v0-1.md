# AetherPress Internals (v0.1)

- **Document Version:** dv00
- **Datetime:** 2025-10-10
- **Branch:** aether-rewrite/client-phase2-AAA

This document provides a brief, high-level overview of the AetherPress application's architecture as of the `client-v2` implementation.

## Overview

AetherPress is a web application that allows users to submit a text prompt to a backend service and receive a generated content preview in return. The system is composed of a Svelte frontend (`client-v2`) and a Node.js/Express backend (`server`).

## Backend Orchestration: `genieService`

**Note:** `genieService` is the supervisor, and the application services are its subordinates.

The `genieService.js` file is the "brain" of the backend's content generation flow. It is a robust, fault-tolerant orchestrator with the following priorities:

1.  **Handle Demos**: It uses a `helloWorldService` for simple "hello" prompts as a shortcut for demos or testing.
2.  **Use Real AI**: It attempts to use the primary `aiService` for all other prompts, which makes calls to an external AI provider (e.g., Gemini).
3.  **Fallback Gracefully**: If the AI service isn't installed or configured, it uses a `sampleService` to provide a deterministic, pre-canned experience.
4.  **Propagate Real Errors**: If the AI service is configured but fails during runtime, it allows the error to surface so it can be debugged.
5.  **Ensure Security and Consistency**: It sanitizes all HTML output to prevent XSS attacks and standardizes the data structure before returning it, ensuring the rest of the system receives clean, predictable data.

## Frontend-Backend API Contract

The client and server communicate over a simple API endpoint managed by `client-v2/src/lib/promptStore.js`.

- **Endpoint**: `POST /prompt`
- **Frontend Sends**: A JSON object with the user's prompt:
  ```json
  { "prompt": "A story about a robot..." }
  ```
- **Backend Responds**: A JSON object containing the generated HTML preview. The frontend is designed to find the HTML in several possible fields (`preview`, `data.preview`, `data.content.html`) for flexibility.
  ```json
  {
    "success": true,
    "preview": "<html>...</html>"
  }
  ```
- **Interaction**: The process is asynchronous. The frontend UI enters a loading state while it awaits the server's response.

## Frontend Application Structure

The frontend is a single-page application built with Svelte and Vite. Its primary purpose is to provide a UI for submitting prompts and viewing the generated content.

- **`App.svelte`**: The root component that lays out the main UI. It orchestrates the other components and manages the overall state (e.g., loading, errors).
- **`PromptForm.svelte`**: A form with a text area for user input and a submit button. It emits a `submit` event with the prompt text.
- **`PreviewWindow.svelte`**: The area where the final, generated HTML from the server is rendered for the user to see.
- **`promptStore.js`**: A Svelte store that manages the application's state (the current prompt, loading status, errors) and handles the API call to the backend.

### End-to-End Example: The 'hello' Prompt

Here is the step-by-step flow of what happens when a user enters 'hello' and clicks the generate button.

1.  **User Action**: The user types `hello` into the text area in the `PromptForm.svelte` component and clicks the submit button.

2.  **Frontend Request**:

    - The `promptStore` on the client is called. It immediately sets its state to `loading: true`, causing the UI to show a loading indicator.
    - It sends a `POST` request to the `/prompt` endpoint on the server, with the JSON body: `{ "prompt": "hello" }`.

3.  **Backend Processing**:

    - The Node.js server routes the request to `genieService.js`.
    - `genieService` checks the prompt text. It sees that the prompt contains the word "hello" and that no other custom options were selected.
    - This triggers a special **shortcut**. Instead of calling the main AI service, it calls the local `helloWorldService`.

4.  **Backend Response**:

    - `helloWorldService` returns a pre-defined, static content payload (e.g., a simple "Hello World" title and body).
    - `genieService` receives this content, sanitizes it for security, and packages it into the standard JSON response format.
    - The server sends this JSON back to the client.

5.  **Frontend Display**:

    - The client's `promptStore` receives the response from the server.
    - It extracts the HTML content from the JSON payload.
    - It updates a shared `previewStore` with this HTML and sets its state to `loading: false`.

6.  **Final Result**: The `PreviewWindow.svelte` component, which is listening for changes to the `previewStore`, automatically updates and renders the simple, static HTML that was generated by the `helloWorldService`. The user sees a basic "Hello World" message in the preview area.

## Investigation: Why the 'hello' Prompt Fails

The end-to-end flow described above is logical, but it does not match the observed reality. In practice, when the user submits the 'hello' prompt, the `PreviewWindow` remains empty. This section provides a preliminary analysis of why this failure occurs.

The root of the problem lies not in the high-level orchestration, but in the specific data returned by the final service in the chain: `helloWorldService.js`.

### Analysis of the Failure

1.  **The Flaw is in the Payload**: The `genieService` correctly identifies the 'hello' prompt and calls the `helloWorldService` as expected. The entire server-side and client-side logic for handling the request and response is sound.

2.  **A Broken Contract**: The failure occurs because the `helloWorldService` does not return the data structure that its consumer, `genieService`, expects. While `genieService` anticipates receiving an object with a `content.body` or `content.html` property containing the HTML to be rendered, the `helloWorldService` is likely returning an object where this property is either missing, `null`, or an empty string.

3.  **The Ripple Effect**:
    - `genieService` receives this malformed response from `helloWorldService`.
    - When the main `/prompt` handler in `index.js` attempts to extract the HTML to create the user-facing `preview` property, it finds `null`.
    - The server then sends a perfectly valid JSON response to the client, but the value of the `preview` key is `null`.
    - The client's `promptStore` correctly parses the JSON, extracts the `null` value for the HTML, and updates the `previewStore`.
    - The `PreviewWindow` component receives the `null` value and, correctly, renders nothing.

### Conclusion of Initial Investigation

The bug is not a logic error in the application's control flow but a data contract violation. The `helloWorldService.js` is the source of the failure. It is not fulfilling its contract to provide valid, renderable HTML content to the `genieService`.

## Deeper Investigation: The Flaw in Orchestration

The previous analysis correctly identified the data from `helloWorldService` as the origin of the problem, but a deeper look reveals the true issue lies in a failure of orchestration within `genieService` itself.

The core architectural principle is that `genieService` is the sole orchestrator. All other services (`helloWorldService`, `aiService`, `sampleService`) are application services that report to it. `genieService`'s primary responsibility is to manage these services and **ensure it provides a single, consistent data structure** to its own caller (the route handler in `index.js`).

### The Divergence within `genieService`

The failure occurs because `genieService` violates this principle. Inside its `generate` function, it has two mutually exclusive code paths:

1.  **The 'hello' Path**: An `if` block that calls `helloWorldService`.
2.  **The Main Path**: A subsequent block of code that calls `aiService` (or `sampleService` as a fallback).

The critical flaw is that these two paths do not converge into a single, common processing step before returning. The 'hello' path takes the result from `helloWorldService`, performs some sanitation, and returns immediately. The Main Path does the same for the `aiService`.

This creates two slightly different data shapes. The orchestrator is supposed to abstract away the differences between `helloWorldService` and `aiService`, but due to its divergent internal logic, it is instead leaking those differences up to the route handler.

### Conclusion of Deeper Investigation

The true bug is an architectural inconsistency within `genieService`. It fails its duty as an orchestrator by not guaranteeing a uniform response structure from all its internal code paths. The problem isn't that `helloWorldService` is "bad," but that `genieService` doesn't process its result through a common, final processing step to match the structure it produces for all other cases. This leads to the downstream error where the final `preview` property is not correctly generated for the 'hello' path.

### Recommended Solution: Refactor `genieService`

The recommended solution is to refactor `genieService` to properly converge its internal code paths, ensuring it always produces a consistent output structure.

#### Implementation Scope (Solution 1)

1.  **Select the Service**: The `generate` function will first have a selection block that decides which application service to call (`helloWorldService`, `aiService`, or `sampleService` as a fallback). The result from the chosen service will be stored in a single, common variable (e.g., `serviceResult`).

2.  **Common Processing Block**: After the selection block, a single, shared block of code will process the `serviceResult` object, regardless of which service it came from. This block will be responsible for all common tasks:

    - Sanitizing the HTML content.
    - Converting `persistIntents` into `persistInstructions`.
    - Assembling the final, standardized `data` object for the return value.

3.  **Single Return**: A single `return` statement at the end of the function will return the consistently structured object.

This approach corrects the architectural flaw, makes the code more maintainable and less repetitive, and ensures that `genieService` properly fulfills its role as an orchestrator.

## Troubleshooting the 'hello' Prompt Failure

Even after the architectural improvements, manual testing revealed that the frontend preview area was still not updating. This prompted a deeper investigation.

### Summary of Investigation

1.  **Initial Fix**: The `genieService` was refactored to ensure it produced a consistent data structure, correcting the initial architectural flaw. This was implemented in the `...-sol1` branch.
2.  **Second Finding (404 Error)**: Using browser developer tools, we discovered the client was sending requests to itself (`:5174/prompt`) instead of the backend. This was due to a missing proxy configuration.
3.  **Second Fix**: A proxy was added to `client-v2/vite.config.js` to correctly forward requests from the frontend dev server to the backend server (`:3000`).
4.  **Current Status**: Despite both fixes, the UI still does not update. The request is now successfully reaching the backend, but no successful response is being returned to the client. This strongly indicates a **server-side runtime error**.

### Current Hypothesis

The error most likely occurs in a step _after_ content generation but _before_ the final response is sent. Based on the code flow in `server/index.js`, the `persistence.execute()` service is the prime suspect. It is called after `genieService` returns but before `res.json()` is called. A failure in this persistence step would crash the request and prevent the user from ever seeing their preview.

## Proposed Solution: Decouple Preview from Persistence

The user's primary goal is to get a preview of the generated content quickly. The persistence of that content is a secondary, background task that should not block the user experience.

### Design

The solution is to modify the `POST /prompt` handler in `server/index.js` to decouple these two steps.

1.  **Generate and Respond Immediately**: The handler should call `genieService` to get the content, assemble the response payload, and send it back to the client **immediately**.

2.  **Persist Asynchronously**: After the response has been sent, the handler will then, asynchronously, call `persistence.execute()`. This can be done with a simple `await` or by wrapping it in a `setTimeout(..., 0)` to ensure the event loop sends the response first.

### Benefits

- **Improves User Experience**: The user gets their preview instantly, regardless of whether the database write succeeds or fails.
- **Increases Robustness**: A failure in the persistence layer (a common source of errors) will no longer crash the user-facing request. The failure can be logged on the server for maintenance without impacting the user.
- **Aligns with Intent**: This change better reflects the application's priorities: preview first, persistence second.
